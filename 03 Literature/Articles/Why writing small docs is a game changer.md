# Why writing small docs is a game changer

> [!abstract] Summary
> Pursue small docs just as we pursue small commits. They reduce cognitive load on reader and writer by focusing on a single idea at a time. Doing this along with consistent organisation can break down a big writing project into its many manageable pieces.
## Why

1. **Quicker reviews.** Small docs don't take as long to read so getting feedback is much quicker.
2. **Thorough feedback.** A focused doc reduces reading fatigue. It's easier to narrow in on the content without being overwhelmed.
3. **Lower stakes.** if a doc is off track there's less wasted effort when scraping it.
4. **Easier integration.** Less likely to conflict with parallel doc work.
5. **Simpler to polish.** Refining a small doc is more manageable.
6. **Unblocked workflows.** When submitting one small doc for review you can continue work on another one.
7. **Flexible abandonment.** If a project changes direction it's easier to set aside several small docs than untangle one large one.
## How

1. **Minimal argument.** Focus on one self contained idea in each doc. E.g. one feature or proposal, err on the side writing less.
2. **Self contained text.** Include al necessary background information in the doc so the reader doesn't need to jump around.
3. **Not overly simple.** Don't make the doc so small as to leave the idea unclear. E.g. include examples for API usage.
4. **Constrained.** Keep each doc under 3 pages. Long docs may need to be broken down to fit this criteria.
5. **Write in parallel.** Work on several docs at once. While one is being reviewed you can continue work on the other.
6. **Draft background docs.** If your main doc requires understanding of different and existing ideas create a separate background doc first. This may be used by several other docs to set the stage. Think of this as a concept guide or overview.
7. **Organise effectively.** Without good organization small docs can become fragmented and confusing. Prevent this with clear categories, tags, and libraries. Good information architecture is key.
8. **Link related content.** Include links directly in the small doc. These can be links to background docs, relevant proposals, further context, and so on.
9. **Maintain hygiene.** Regularly clean and manage your docs. Update outdates docs and deprecate irrelevant ones.

> [!quote]
> As technical writers, we aim to make information clear, accessible, and actionable. But there’s another factor to consider: how the size of a document impacts its effectiveness. Small, focused docs can save time, reduce stress, and make the review process much smoother.
